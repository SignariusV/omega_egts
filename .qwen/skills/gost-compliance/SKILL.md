# ГОСТ Compliance — Проверка соответствия ГОСТ 33465/33464

## Назначение

Навык для проверки соответствия кода **серверного тестера УСВ (OMEGA_EGTS)** требованиям:
- **ГОСТ 33465-2015** — Протокол обмена данными УСВ с инфраструктурой (транспортный уровень)
- **ГОСТ 33464** — Система экстренного реагирования при авариях (требования к УСВ)
- **ГОСТ 33465-2023** (будущая поддержка) — Актуальная редакция

## Когда использовать

- Ревью кода на соответствие протоколу EGTS
- Проверка структуры пакетов (TL/SL, заголовки, CRC)
- Валидация таймаутов и параметров протокола
- Проверка FSM состояний авторизации
- Анализ сценариев тестирования на полноту покрытия ГОСТ
- Ревью изменений в ядре EGTS-протокола

## Инструкции по проверке

### 1. Транспортный уровень (ГОСТ 33465)

При ревью кода проверяй:

**Структура пакета EGTS:**
- [ ] Поле `PR` (признаки обработки) — биты: prio, archive, compress, reroute, ack, reply
- [ ] Поле `HL` (длина дополнительного заголовка) — корректность вычисления
- [ ] CRC-16 проверка (`CRC16(X16+X15+X8+1)`)
- [ ] Адресация (`ADDR`): тип (ANUM/AROUT), значение, направление
- [ ] Идентификатор соединения (`CID`)
- [ ] Номер пакета (`PN`) и подтверждение (`RP`)

**Таймауты и параметры:**
- [ ] `TL_RESPONSE_TO` = **5 с** — ожидание подтверждения
- [ ] `TL_RESEND_ATTEMPTS` = **3** — повторные попытки отправки
- [ ] `TL_RECONNECT_TO` = **30 с** — повторное соединение
- [ ] `EGTS_SL_NOT_AUTH_TO` = **6 с** — таймаут авторизации на сервисном уровне
- [ ] `CMW_SCPI_TIMEOUT` = **5 с** — таймаут SCPI-команды к CMW-500
- [ ] `CMW_SCPI_RETRIES` = **3** — повторные попытки SCPI

**Обработка ошибок:**
- [ ] 100% пакетов логируется (включая CRC-ошибки)
- [ ] Лог содержит hex + parsed представление каждого пакета
- [ ] RESPONSE отправляется в течение TL_RESPONSE_TO
- [ ] При отсутствии подтверждения — TL_RESEND_ATTEMPTS повторов

### 2. Сервисный уровень (EGTS-сообщения)

При проверке сценариев валидируй наличие и порядок:

**Авторизация:**
```
TERM_IDENTITY → AUTH_PARAMS → AUTH_INFO → RESULT_CODE
```
- [ ] TERM_IDENTITY содержит `TERM_CODE` (код терминала)
- [ ] AUTH_PARAMS запрашивает параметры аутентификации
- [ ] AUTH_INFO передаёт учётные данные
- [ ] RESULT_CODE подтверждает/отклоняет авторизацию

**Телеметрия:**
- [ ] NAV_DATA содержит координаты (широта, долгота), скорость, курс
- [ ] Точность координат соответствует ГОСТ
- [ ] Временные метки в UTC

**Траектория:**
- [ ] TRACK_DATA — последовательность точек с координатами и временем
- [ ] Корректный порядок следования
- [ ] Интервалы между точками соответствуют настройкам

**Профиль ускорения:**
- [ ] ACCEL_DATA с разрешением ≤ **0.01G**
- [ ] Поддержка ASI15 (15-битное представление)
- [ ] Временные метки синхронизированы

**eCall:**
- [ ] RAW_MSD_DATA содержит ASN.1-структуру MSD
- [ ] МНД (минимальный набор данных) соответствует ГОСТ
- [ ] Поля: VIN, координаты, время, тип ДТП, кол-во пассажиров

**Обновление ПО:**
- [ ] SERVICE_PART_DATA — часть ПО
- [ ] SERVICE_FULL_DATA — полный образ
- [ ] ODH (On-Demand Handshake) подтверждение

### 3. Архитектурные требования (из ТЗ)

**Компоненты:**
- [ ] Компоненты общаются **только** через EventBus (3 типа событий, не больше)
- [ ] Парсинг ≠ валидация ≠ бизнес-логика ≠ логирование ≠ интерфейс (разделены)
- [ ] FSM тестируется изолированно (без сети, CMW-500, EGTS)
- [ ] Ядро работает автономно — без сценария принимает пакеты, отправляет RESPONSE, логирует

**Технологии:**
- [ ] Python >= 3.12
- [ ] asyncio для TCP
- [ ] PyVISA для SCPI/VISA over LAN
- [ ] pytest для тестов, покрытие ≥ 90%

**Принципы:**
- [ ] Единое ядро, CLI/GUI — тонкие обёртки
- [ ] Без DI-фреймворков — ручной DI через конструкторы
- [ ] Без абстракций «на вырост»
- [ ] Без ORM — JSON-файл для учётных данных

### 4. Сценарии тестирования (полнота покрытия)

При проверке сценариев убеждайся в наличии всех 10:

| # | Сценарий | Ключевые сообщения |
|---|----------|-------------------|
| 1 | Авторизация | TERM_IDENTITY, AUTH_PARAMS, AUTH_INFO, RESULT_CODE |
| 2 | Телеметрия | NAV_DATA, координаты, скорость |
| 3 | Траектория | TRACK_DATA, последовательность точек |
| 4 | Профиль ускорения | ACCEL_DATA, ≤0.01G, ASI15 |
| 5 | eCall | RAW_MSD_DATA, ASN.1 MSD, МНД |
| 6 | Обновление ПО | SERVICE_PART_DATA, SERVICE_FULL_DATA, ODH |
| 7 | Команды и параметры | COMMAND_DATA, запрос/установка параметров |
| 8 | Режим тестирования | TEST_MODE_ON/OFF, проверочные пакеты |
| 9 | SMS-канал | Приём/отправка EGTS через SMS, конкатенация |
| 10 | Пассивный режим | TID=0, SMS с учётными данными, повторная авторизация |

## Примеры

### Пример 1: Проверка таймаутов
```
Пользователь: Проверь таймауты в tcp_server.py
→ Загрузи этот навык, проверь TL_RESPONSE_TO=5с, TL_RESEND_ATTEMPTS=3, TL_RECONNECT_TO=30с
→ Выдай отчёт с указанием найденных/отсутствующих таймаутов
```

### Пример 2: Ревью EGTS-пакета
```
Пользователь: Проверь структуру пакета EGTS в packet_parser.py
→ Загрузи этот навык, проверь PR, HL, CRC-16, ADDR, CID, PN, RP
→ Выдай отчёт с ✅/⚠️/❌ по каждому полю
```

### Пример 3: Проверка сценария авторизации
```
Пользователь: Проверь сценарий авторизации в scenarios/auth.json
→ Загрузи этот навык, проверь порядок: TERM_IDENTITY → AUTH_PARAMS → AUTH_INFO → RESULT_CODE
→ Убедись в наличии всех обязательных полей
```
